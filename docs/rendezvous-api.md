# Rendezvous Server — API Reference

Status: v1
Scope: the HTTP surface of `apps/envoix-server` (Issue #6).
Design rationale: `docs/rendezvous-design.md`. This document is the
client-implementer contract; the design doc explains *why*.

## Base URL and versioning

All endpoints live under `/api/v1/`. The reference deployment is
`https://home.chkxwlyh.com/envoix`, so a full URL looks like:

```
https://home.chkxwlyh.com/envoix/api/v1/sessions
```

The server is deployment-agnostic: clients must treat the base URL as
opaque configuration (it arrives via the QR payload, Issue #5).

Additive changes (new fields, new candidate kinds, new endpoints) can
appear within `/api/v1` at any time — clients must ignore unknown JSON
fields. Breaking changes require `/api/v2`.

## Authentication

Every endpoint except `POST /api/v1/sessions`-registration's body rules
and `GET /api/v1/health` requires:

```
Authorization: Bearer <capability>
```

A capability is exactly 32 lowercase hex characters (128 bits; generate
with a CSPRNG such as `getrandom`). Two role-separated capabilities
exist per session:

| Capability | Held by | Authorises |
|---|---|---|
| `receiver_cap` | receiver | register, publish receiver candidates, poll, close |
| `sender_cap` | sender | join, publish sender candidates, poll |

The server stores only BLAKE3 hashes. A missing or malformed
`Authorization` header is `401 unauthorized` — the session id in the
URL is not inspected first, so the endpoint cannot be used to probe for
live session ids.

## Error envelope

Every non-2xx response carries:

```json
{ "code": "session_expired", "message": "human-readable detail" }
```

`code` is stable and machine-parseable; `message` may change freely.

| `code` | HTTP | Meaning | Client action |
|---|---|---|---|
| `invalid_request` | 400 | Malformed payload / field out of bounds | Fix the request |
| `unauthorized` | 401 | Capability missing or mismatched | Re-check pairing data |
| `session_not_found` | 404 | Id was never registered | Re-pair via QR |
| `session_expired` | 404 | Session existed; TTL elapsed | Re-pair via QR |
| `conflict` | 409 | Duplicate registration of the same id | Generate a new id |
| `session_closed` | 409 | Receiver explicitly closed the session | Stop polling; surface "transfer ended" |
| `payload_too_large` | 413 | Body exceeded 64 KiB | Fix the request |
| `unsupported_version` | 422 | No common protocol version | Upgrade client |
| `capacity_exceeded` | 503 | Server at session cap | Retry with backoff |
| `service_shutting_down` | 503 | Server draining for restart | Retry with backoff |
| `internal` | 500 | Server bug; details in server logs | Retry once, then report |

## Endpoints

### POST /api/v1/sessions — register (receiver)

Headers: `Authorization: Bearer <receiver_cap>`

```json
{
  "session_id": "0123456789abcdef0123456789abcdef",
  "sender_cap_hash": "<64 lowercase hex chars — BLAKE3 of sender_cap>",
  "peer_metadata": {
    "protocol_versions": [1],
    "strategies": ["lan", "ipv6"]
  },
  "ttl_seconds": 300
}
```

- `session_id` — 32 lowercase hex chars, generated by the receiver. Must
  differ from both capabilities.
- `sender_cap_hash` — the receiver hashes `sender_cap` locally; the raw
  sender capability never reaches the server.
- `ttl_seconds` — optional. Default 300, clamped server-side to 1800.
- `strategies` — optional, advisory, passed through to the peer.

`201 Created`:

```json
{
  "session_id": "0123456789abcdef0123456789abcdef",
  "expires_at": "2026-06-09T15:20:00Z",
  "probe_token": "<114 lowercase hex chars>",
  "probe_endpoints": ["203.0.113.1:9101", "203.0.113.1:9102"]
}
```

- `expires_at` reflects the *effective* TTL after clamping.
- `probe_token` / `probe_endpoints` — present **only when the server's
  reflexive probe service is enabled**; absence means the feature is
  off, not an error. See "UDP reflexive probe" below. The token is
  opaque: store it, echo it in probes, never parse it.

### POST /api/v1/sessions/{id}/join — join (sender)

Headers: `Authorization: Bearer <sender_cap>`

```json
{ "peer_metadata": { "protocol_versions": [1], "strategies": ["lan"] } }
```

`200 OK`:

```json
{
  "expires_at": "2026-06-09T15:21:30Z",
  "probe_token": "<114 lowercase hex chars>",
  "probe_endpoints": ["203.0.113.1:9101", "203.0.113.1:9102"]
}
```

Joining twice is idempotent (refreshes metadata and TTL). The sender
gets its own `probe_token` (role-bound; tokens are not interchangeable
between peers); the probe fields follow the same present-only-when-
enabled rule as registration.

### POST /api/v1/sessions/{id}/candidates — publish

Headers: either capability. The candidate is attributed to the role the
capability proves.

```json
{
  "kind": "host",
  "transport": "quic",
  "addr": "10.0.0.5:9000",
  "priority": 100
}
```

- `kind` — `"host"` (LAN-local), `"ipv6_global"` (publicly routable
  IPv6), or `"server_reflexive_udp"` (a NAT mapping learned via the
  probe service or public STUN). Unknown kinds are rejected with `400`.
  `ipv6_global` requires an IPv6 `addr`.
- `transport` — `"quic"`.
- `addr` — `"ip:port"` for IPv4, `"[ip]:port"` for IPv6.
- `priority` — optional advisory hint, higher first, default 0.

`200 OK` with the canonical stored record:

```json
{
  "kind": "host",
  "transport": "quic",
  "addr": "10.0.0.5:9000",
  "priority": 100,
  "sequence": 3,
  "published_at": "2026-06-09T15:14:12Z"
}
```

Re-publishing an identical `(kind, transport, addr)` is a no-op: the
response is the existing record with its original `sequence`. Each
session holds at most 32 candidates per peer (then `400`).

### GET /api/v1/sessions/{id}/candidates?since=N — poll

Headers: either capability. Returns the **other** peer's state.

`since` — optional, default 0. Only candidates with `sequence > N` are
returned; pass the highest `sequence` you have seen.

`200 OK`:

```json
{
  "peer_metadata": {
    "observed_http_addr": "203.0.113.10:0",
    "protocol_versions": [1],
    "strategies": ["lan", "ipv6"],
    "first_seen": "2026-06-09T15:14:09Z",
    "last_seen": "2026-06-09T15:14:32Z"
  },
  "candidates": [
    {
      "kind": "ipv6_global",
      "transport": "quic",
      "addr": "[2001:db8::1]:9000",
      "priority": 100,
      "sequence": 1,
      "published_at": "2026-06-09T15:14:12Z"
    }
  ]
}
```

- `peer_metadata` is `null` while the receiver polls a session the
  sender has not joined yet — its appearance is the join signal.
- `observed_http_addr` is **advisory diagnostics only** (HTTP source
  address as seen through the reverse proxy; port is 0 = unknown). It
  is *not* a QUIC candidate — do not connect to it.
- An empty `candidates` array is the normal "nothing new" answer; the
  server returns immediately (short-poll). Recommended client polling
  interval: 200–500 ms (non-normative).

### UDP reflexive probe — discover your NAT mapping

Not HTTP: a raw UDP exchange against the addresses in
`probe_endpoints`. Full rationale in
`docs/reflexive-discovery-design.md`; this is the client contract.

**Send the probe from the same UDP socket your QUIC endpoint uses.**
NAT mappings are per-socket — a mapping learned on any other socket is
useless for the later connection.

Request — exactly 70 bytes:

```
offset  size  field
0       4     magic    = 3f 45 56 58
4       1     version  = 01
5       8     txid     — random bytes you choose; replies echo them
13      57    token    — probe_token hex-decoded, verbatim
```

Reply — 20 bytes (IPv4) or 32 bytes (IPv6), sent to the probe's source:

```
offset  size   field
0       4      magic (same)
4       1      version
5       8      txid (echoed — discard replies that match nothing in flight)
13      1      family   = 01 IPv4 / 02 IPv6
14      2      xport    = your observed port XOR 0x3f45
16      4|16   xaddr    = your observed addr XOR keystream
                          keystream = magic ‖ txid ‖ magic (16 bytes)
```

The XOR is not encryption — it stops NAT ALGs from rewriting address
bytes they recognise. Decode: `addr[i] = xaddr[i] ^ keystream[i]`,
`port = xport ^ 0x3f45`.

Client rules:

- Retransmit on silence: send at 0 / 500 / 1500 / 3500 / 7500 ms, then
  give up (RFC 8489 §6.2.1 schedule). Silence means an invalid/expired
  token, a dead session, or packet loss — the server never explains.
- Probe **both** endpoints in `probe_endpoints` from the same socket
  and compare the two observed addresses: **equal** ⇒ your NAT mapping
  is endpoint-independent and the address is usable for hole punching;
  **different** ⇒ symmetric NAT/CGNAT — skip punching, proceed to
  relay fallback.
- You do not need to publish the result: the server auto-publishes
  your observed mapping as a `server_reflexive_udp` candidate
  (priority 50) the moment a valid probe arrives. The peer sees it on
  its next poll. Probing twice is harmless (duplicate candidates
  dedup).
- The reply tells *you* your own mapping — keep it for the
  symmetric-NAT comparison and diagnostics.

### DELETE /api/v1/sessions/{id} — close (receiver only)

Headers: `Authorization: Bearer <receiver_cap>`

`204 No Content`. The sender's capability gets `401`. Subsequent
requests against the session return `409 session_closed` for roughly
one TTL cycle, then `404 session_not_found`.

### GET /api/v1/health

Unauthenticated. `200 OK`, body `ok`. Liveness only.

### GET /api/v1/stats — operator only

`Authorization: Bearer <admin-token>` where the token is the server's
`--admin-token` / `ENVOIX_ADMIN_TOKEN` setting. When no admin token is
configured the route answers plain `404`.

`200 OK`:

```json
{
  "uptime_seconds": 12345,
  "sessions": {
    "active": 8,
    "created_total": 142,
    "expired_total": 117,
    "closed_total": 17,
    "rejected_capacity_total": 0,
    "rejected_authz_total": 3
  },
  "candidates": { "published_total": 318, "active": 24 },
  "probes": { "received_total": 12, "invalid_total": 2, "published_total": 9 },
  "requests": { "total": 1487, "by_status": { "200": 1431, "401": 3 } }
}
```

## Typical flow

```
receiver                         server                          sender
   │ POST /sessions ──────────────►│
   │ ◄─────────────── 201 created  │
   │   (QR: url, session_id, sender_cap)  ─────────────────────────►
   │ POST /sessions/{id}/candidates│◄─────── POST /sessions/{id}/join
   │ GET  /sessions/{id}/candidates│◄── POST /sessions/{id}/candidates
   │          (poll loop)          │◄── GET /sessions/{id}/candidates
   │                               │            (poll loop)
   │      ── QUIC connection attempt happens peer-to-peer ──
   │ DELETE /sessions/{id} ───────►│
```

The server never carries file bytes; once a QUIC connection is up the
peers run end-to-end authentication (SPAKE2, `docs/auth.md`) and the
transfer protocol directly.
